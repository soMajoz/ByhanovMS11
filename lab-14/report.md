# Отчет по лабораторной работе №14

## Часть 1. PostgreSQL

Цель части: создать реляционную схему интернет-магазина, заполнить таблицы тестовыми данными, написать аналитические SQL-запросы и показать оптимизацию через индексы.

В `part1-postgresql/schema.sql` реализованы таблицы `customers`, `products`, `orders`, `order_items` с первичными и внешними ключами, проверками статуса заказа, способа оплаты, положительных цен и количества. Данные покрывают покупателей из разных городов, товары из категорий `Электроника`, `Книги`, `Канцелярия`, оплаченные, отправленные, созданные и отмененные заказы.

В `queries.sql` реализованы три обязательных аналитических запроса:

- выручка и количество продаж по категориям;
- топ-3 клиента по сумме оплаченных/отправленных заказов;
- динамика выручки по месяцам.

В `index_analysis.sql` добавлены индексы по статусу и дате заказа, покупателю, связям позиций заказа и категории товара. Там же подготовлен `EXPLAIN ANALYZE` до и после создания индексов.

## Часть 2. MongoDB

Цель части: реализовать ту же предметную область в документной модели MongoDB, выполнить CRUD-операции, агрегации и сравнить подход с PostgreSQL.

В `part2-mongodb/init.js` заказы хранятся документами с вложенными `customer` и `items`, что уменьшает потребность в join-like операциях. В `queries.js` реализованы создание заказа, чтение последних заказов, обновление статуса, удаление отмененных заказов, выручка по категориям и топ клиентов. В `comparison.js` показан аналог SQL-запроса месячной выручки, а `indexes.js` создает индексы по статусу/дате, покупателю, категории вложенных товаров и уникальному номеру заказа.

## Сравнение PostgreSQL и MongoDB

| Критерий | PostgreSQL | MongoDB |
|---|---|---|
| Модель | Нормализованные таблицы и связи | Документы заказов с вложенными позициями |
| Аналитика | SQL, `JOIN`, `GROUP BY`, строгие типы | Aggregation pipeline, `$unwind`, `$group` |
| Целостность | Внешние ключи и `CHECK`-ограничения | Контроль в приложении и индексах |
| Простые чтения заказа | Нужно соединять таблицы | Заказ читается одним документом |
| Сложная отчетность | Более привычна для BI/SQL | Удобна при правильно выбранной структуре документов |
