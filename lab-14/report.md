## Ответы на контрольные вопросы (Часть 1)

1. **В чем разница между `WHERE` и `HAVING` в агрегатных запросах?**  
   `WHERE` фильтрует строки до этапа группировки, а `HAVING` фильтрует уже сформированные группы после выполнения `GROUP BY`. Поэтому условия по агрегатам (`SUM`, `COUNT`, `AVG`) задаются именно в `HAVING`.

2. **Почему при `GROUP BY` нужно перечислять все неагрегированные столбцы?**  
   Потому что SQL должен однозначно понимать, по каким признакам объединять строки в группы. Если столбец выводится в `SELECT`, но не участвует в агрегатной функции, он должен входить в `GROUP BY`, иначе результат был бы неоднозначным.

3. **Как изменяется `EXPLAIN ANALYZE` после добавления индекса? Что означают `Seq Scan` и `Index Scan`?**  
   После добавления индекса оптимизатор получает возможность выбирать более быстрый способ доступа к строкам. `Seq Scan` означает последовательное чтение всей таблицы, а `Index Scan` - использование индексной структуры для поиска нужных записей. На больших таблицах это особенно важно, поскольку индекс позволяет резко сократить объем читаемых данных.

Такой подход лучше подходит для сценариев, когда система часто читает заказ целиком как единый объект.

## Ответы на контрольные вопросы (Часть 2)

1. **Как в MongoDB представлена связь "заказ-товары" в отличие от PostgreSQL?**  
   В PostgreSQL связь реализована через отдельную таблицу `order_items`, связанную с заказом внешним ключом. В MongoDB позиции заказа вложены прямо в документ заказа в массив `items`. Это делает чтение заказа целиком проще, но меняет способ выполнения аналитических запросов.

2. **В каком случае документная модель удобнее реляционной?**  
   Документная модель удобнее тогда, когда чаще всего требуется читать объект целиком, например заказ вместе с вложенными товарами и данными клиента. В данной лабораторной это видно на примере чтения одного заказа, который в MongoDB извлекается одним документом, а в PostgreSQL собирается из нескольких таблиц.

3. **Какие операции в MongoDB оказались проще, а какие сложнее по сравнению с SQL?**  
   Проще оказались операции чтения и обновления одного заказа целиком, а также сценарии, где данные удобно хранить в одном документе. Сложнее оказались аналитические выборки, поскольку вместо привычного SQL пришлось строить многошаговые aggregation pipeline с `$lookup`, `$unwind`, `$group` и `$project`.

4. **Что такое `$unwind` и зачем он нужен?**  
   `$unwind` разворачивает массив внутри документа в набор отдельных записей для последующей обработки. В этой работе он был нужен для того, чтобы анализировать позиции товаров внутри массива `items` так, как будто это отдельные строки.

## Сравнение PostgreSQL и MongoDB

| Характеристика | PostgreSQL | MongoDB |
|----------------|------------|---------|
| Схема | Жестко заданная | Гибкая |
| Связи | Внешние ключи | Вложенные документы / ссылки |
| Аналитика | SQL, `JOIN`, `GROUP BY` | Aggregation pipeline |
| Чтение одного заказа | Через несколько таблиц | Одним документом |
| Контроль целостности | Высокий, на уровне СУБД | Выше на уровне приложения |
| Гибкость структуры | Ниже | Выше |

Практически лабораторная показала, что PostgreSQL удобнее для строгой отчетности и транзакционной модели, а MongoDB удобнее для хранения агрегированного объекта заказа и гибкой эволюции схемы.
