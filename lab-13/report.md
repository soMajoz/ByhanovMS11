## Часть 1. Статический анализ с Bandit

### Исходные уязвимости Flask-приложения
В ходе анализа предоставленного кода на Python (Flask) были обнаружены следующие критичные проблемы безопасности:
1. **SQL-инъекция** в обработчиках `/user` и `/search`, где данные напрямую подставлялись в строку SQL-запроса через f-строки.
2. **Command Injection** в эндпоинте `/execute` при передаче пользовательского ввода напрямую в `subprocess.check_output(cmd, shell=True)`.
3. Зашитый (Hardcoded) глобальный API ключ.
4. Открытый `debug=True` и `host=0.0.0.0` на продакшене.

### Результат сканирования до модификаций
Bandit выявил:
- `[B608:hardcoded_sql_expressions]` (High): Использование потенциально небезопасных SQL-выражений.
- `[B602:subprocess_popen_with_shell_equals_true]` (High): Выполнение команды с флагом shell.
- `[B105:hardcoded_password_string]` (Low/Medium): Хардкод пароля (в данном случае API ключа).
- `[B201:flask_debug_true]` (High): Приложение Flask запущено с флагом `debug=True`.

### Исправления
Для исправления SQL-инъекций был применен метод параметризованных запросов SQL:
```python
query = "SELECT * FROM users WHERE id = ?"
cursor.execute(query, (user_id,))
```
Для предотвращения RCE (Command Injection) был убран флаг `shell=True`, добавлен `allow-list` и парсинг аргументов:
```python
ALLOWED_COMMANDS = ['echo', 'date', 'whoami']
parts = cmd.split()
if not parts or parts[0] not in ALLOWED_COMMANDS:
    return jsonify({"error": "Command not allowed"}), 403
result = subprocess.check_output(parts)
```
Секретный ключ вынесен в переменные окружения: `API_KEY = os.environ.get('API_KEY')`.

## Ответы на контрольные вопросы (Часть 1)

1. **Какие уязвимости обнаружил Bandit?**
   Bandit обнаружил харкдод-пароли (`B105`), выполнение команд через shell-перехватчик с нефильтрованным вводом (`B602`), небезопасные f-строки в SQL (`B608`), конфигурацию `debug=True` и небезопасный бинд на `0.0.0.0` (`B104`, `B201`).
2. **Почему параметризованные запросы защищают от SQL-инъекций?**
   Суть параметризованного запроса в том, что СУБД компилирует (подготавливает) SQL-код до его конкатенации с данными пользователя, и расценивает любые переданные параметры как строковые литералы (данные), а не как часть исполняемого оператора (кода). Таким образом, логика запроса не меняется.
3. **В чем разница между `subprocess.run(cmd, shell=True)` и `subprocess.run(cmd.split())`?**
   Первый вариант передает всю строку системной оболочке (`/bin/sh` или `cmd.exe`), что позволяет выполнять спецсимволы, такие как `;`, `|`, `&&`. Второй вариант вызывает процесс напрямую и передает ему массив аргументов, оболочка не участвует, следовательно, инъекции спецсимволов не работают.
4. **Какие уязвимости статический анализатор (SAST) не может обнаружить?**
   Проблемы бизнес-логики (например, недоработки контроля доступа Insecure Direct Object Reference, логические дыры в аутентификации), конфигурационные баги реального внешнего сервера и особенности утечек памяти в рантайме.

## Часть 2. Динамическое тестирование и Анализ зависимостей (SCA)

### Описание уязвимого стенда Node.js
В проекте использовались намеренно устаревшие пакеты (`axios@0.18.0`, `ejs@2.6.1`, `express@4.16.0`), также присутствовали явные архитектурные дыры (XSS в EJS шаблонах, SQLi при получении комментариев и SSRF). 

### Результаты SCA
При запуске `npm audit` было выявлено 17 уязвимостей, среди которых: 3 low, 1 moderate, 10 high и 3 critical. Основные угрозы:
- **node-tar** — Arbitrary File Overwrite.
- **axios** — Critical SSRF vulnerability (CVE неактуален в старых версиях).
Автоматическим исправлением (`npm audit fix --force`) были подняты мажорные версии пакетов (например, `ejs` -> `5.0.1`, `axios` -> `1.14.0`), что сократило список уязвимостей до минимума.

### Исправление XSS и внедрение CSP
XSS (межсайтовый скриптинг) в файле `views/index.ejs` был ликвидирован двумя способами:
1. Замена `<%- comment.comment %>` (возвращает сырой HTML-код) на `<%= comment.comment %>` (экранирует спецсимволы по умолчанию в шаблонизаторе EJS).
2. Замена опасной функции `eval()` на сертифицированный парсер `JSON.parse()`.
3. На стороне клиента при добавлении DOM-узлов `innerHTML` был заменен на `document.createElement('div')` + `div.textContent`.

Также в приложении был внедрен Content-Security-Policy заголовок (CSP) для дополнительной блокировки несанкционированных скриптов:
```javascript
const nonce = crypto.randomBytes(16).toString('base64');
res.setHeader('Content-Security-Policy', `default-src 'self'; script-src 'self' 'nonce-${nonce}'; ...`);
```

## Ответы на контрольные вопросы (Часть 2)

1. **Какие CVE были обнаружены в зависимостях?**
   В основном уязвимости в `node-tar` (GHSA-34x7-hfp2-rc4v: Arbitrary File Creation/Overwrite), SSRF в `axios` (устранено в v0.21.1) и проблемы в `http-proxy-agent`.
2. **В чем разница между `npm audit` и `snyk test`?**
   `npm audit` использует стандартную встроенную базу GitHub Advisory Database. `Snyk` имеет собственную, часто более глубокую корпоративную базу данных уязвимостей, возможность отслеживания патчей и детальные рекомендации по исправлению (в том числе для косвенных зависимостей).
3. **Как OWASP ZAP обнаруживает XSS-уязвимости?**
   DAST инструменты вроде ZAP отправляют множество payload'ов (вредоносных строк с JS скриптами) в инпуты, куки и заголовки, после чего анализируют ответный HTTP-документ — если измененный payload вернулся целым в теле HTML и "отрендерился" в исполняемом контексте, регистрируется XSS.
4. **Почему CSP (Content Security Policy) эффективен против XSS?**
   CSP позволяет браузеру понять, с каких серверов (доменов) легально загружать ресурсы, и блокирует Inline-сценарии (если не передан правильный криптографический `nonce` или HASH). Даже если хакеру удалось "просунуть" тег `<script>alert(1)</script>` через экранирование, браузер отклонит его исполнение, так как он нарушает политику.
5. **Какие ограничения есть у DAST-инструментов по сравнению с SAST?**
   DAST видит приложение "снаружи" (как черный ящик), поэтому не может указать на номер строки проблемного кода, не видит часть логики скрытой за авторизацией или сложными капчами, и может упустить уязвимости, которые срабатывают только при специфичных триггерах-стейтах в БД. В отличие от SAST (анализ исходного кода), DAST не имеет 100% покрытия кода, но зато практически лишен "ложных срабатываний" в части реальной уязвимости.
