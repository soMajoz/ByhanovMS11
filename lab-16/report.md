## Часть 1. GigaChat

## Цель части

Показать практическое использование LLM-инструмента для:

- генерации функций;
- рефакторинга плохого кода;
- написания тестов;
- подготовки документации;
- анализа качества результата.

## Что реализовано

Ключевой файл:

- `part1-gigachat/gigachat_client.py`

Он выполняет:

- получение access token;
- отправку запросов в GigaChat API;
- работу с model completion;
- извлечение Python-кода из markdown code block;
- сохранение результатов в локальные файлы.

Это важно, потому что реальный ответ модели не всегда приходит "чистым" кодом. В лабораторной отдельно учтена необходимость разбирать fenced code block и выделять именно исполняемый Python.

## Какие артефакты получены

### `generated_code.py`

Содержит сгенерированные функции:

- `is_prime`
- `fibonacci`
- `normalize_phone`

Эти функции используются как пример задач, которые можно отдавать LLM для автоматической генерации.

### `bad_code.py`

Содержит изначально плохой код с:

- неудачной структурой;
- неявной логикой;
- глобальным состоянием.

### `refactored_code.py`

Содержит переработанный вариант, где:

- введен `OrderItem`;
- добавлен `OrderBasket`;
- валидация сделана явной;
- код стал удобнее тестировать и сопровождать.

### `test_refactored.py`

Содержит pytest-проверки:

- расчет общей суммы корзины;
- поиск товара;
- отклонение невалидного элемента;
- проверку сгенерированных функций.

### `README_generated.md`

Содержит автоматически подготовленную документацию по результату.

### `report/analysis.json`

Содержит сжатый анализ качества итогового кода.

## Анализ качества кода по итогам лабораторной

Файл:

- `part1-gigachat/report/analysis.json`

Содержит две важные группы выводов.

### Сильные стороны generated code

- функции детерминированы;
- есть type hints и docstrings;
- крайние случаи покрыты тестами.

### Риски

- LLM-результат необходимо ревьюить перед production;
- модель может пропускать доменную валидацию.

### Что изменилось после рефакторинга

До:

- глобальный изменяемый список;
- неочевидные короткие имена;
- слабая читаемость.

После:

- `OrderItem` как dataclass;
- `OrderBasket` как отдельная сущность;
- явная валидация и более чистая структура.

Это хороший, предметный пример того, что ИИ полезен как ускоритель, но не заменяет инженерную проверку.

## Реальная проверка Airtable API

Команда:

```text
python lab-16\part2-airtable\export_airtable.py
```

Фактический результат на момент финальной проверки:

```text
403 Client Error: Forbidden for url: https://api.airtable.com/v0/app4cjy5L6ciKl055/Customers?pageSize=100
```

## Как интерпретировать этот результат

Это означает, что:

- `AIRTABLE_BASE_ID` теперь указан в корректном формате `app...`;
- запрос доходит до правильного base endpoint;
- но у текущего `AIRTABLE_TOKEN` недостаточно прав для чтения таблицы `Customers`.

Иными словами, проблема уже не в коде клиента и не в формате base id, а в правах PAT:

- либо база не добавлена в доступ токена;
- либо не выданы нужные scopes, например `data.records:read`;
- либо токен не привязан к нужному workspace/base.

Для сдачи это важно указать честно: Python-клиент и схема готовы, но финальная реальная API-выгрузка зависит от прав доступа в самом Airtable.

## Сравнение low-code и AI-подхода

Лабораторная специально сочетает две разных парадигмы:

### GigaChat

- ускоряет программирование;
- помогает рефакторить и писать тесты;
- требует инженерного ревью и верификации.

### Airtable

- ускоряет сборку бизнес-структуры без большого объема кода;
- удобен для CRUD, форм и визуальных представлений;
- ограничен платформой и настройкой прав доступа.

## Общий вывод

- AI лучше подходит там, где нужен код и логика;
- low-code лучше подходит там, где нужен быстрый рабочий процесс и интерфейс для бизнес-пользователя;
- оба подхода полезны, но каждый имеет свои границы применимости.
